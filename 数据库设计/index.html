🗺数据库设计

字数统计: 2.4k | 阅读时长 ≈ 8

对于任何一个软件项目开发而言，都一定会经历以下的几个阶段：

  • 获取需求阶段
  • 需求分析与业务设计
  • 数据库设计
  • 程序开发与业务实现
  • 程序的测试
  • 程序的运维

数据库设计范式

所有给出的设计范式只能够作为一种参考出现，在初期结构之中可以遵循设计范式进行设计但最终会发现所有的范式都几乎会被打破，只有一个原则： 根据业务的需求尽可能的减少多表查询和复杂查询

第一设计范式(单表)

数据库表中的每个字段都不可再分

Eg:

学生的联系方式可以由地址、邮政编码、电话、email、手机、QQ、微信….. 所以这样的设计就不符合第一设计方式的原则。

Eg(modified):

上表中的字段就不可以再分了，符合第一设计范式

第二设计范式（多对多关系映射）

数据表之中不存在非关键字段对任意一候选关键字段的部分函数依赖
  • 关键点： 函数、函数依赖

函数关系： 数据库表之中的两个列之间不要存在有数学关系。

此表存在以下关系： allprice = unitprice * allprice

依赖： 指通过某几个字段可以成功的推演出唯一的一个字段

Eg(第一种设计方式)

上表设计所存在的问题是没有将学生的选课看成一个独立存在的实体而是直接作为学生实体的属性因此没有考虑【学生实体】【学生所选的课程实体】之间的关系
存在的问题如下：

  • 无法设计一个合适的字段作为主键
  • 课程信息重复，那么如果更新一个课程信息则需要更新N行记录
  • 如何说现在有一门课程一直没人报名参加，则这个课程信息就彻底消失了

如果现在按照可以使用的思路，认为应该设计为三张表，因为一个学生可以参加多门课程，一个课程可以有多个学生参加。

Eg(modified)


CREATE TABLE 学生(
    学生ID    NUMBER PRIMARY KEY ,
    学生姓名    VARCHAR2(20) ,
    学生年龄    NUMBER 
) ;
CREATE TABLE 课程(
    课程ID        NUMBER    PRIMARY KEY ,
    课程名称    VARCHAR2(50) ,
    课程学分    NUMBER 
) ;

CREATE TABLE 学生选课(
    学生ID        NUMBER REFERENCES 学生(学生ID) ,
    课程ID        NUMBER REFERENCES 课程(课程ID) ,
    成绩        NUMBER
) ;
INSERT INTO 学生 VALUES (1,'张三',18) ;
INSERT INTO 学生 VALUES (2,'李四',17) ;
INSERT INTO 课程 VALUES (10,'马克思主义哲学',3) ;
INSERT INTO 课程 VALUES (11,'oracle',1) ;
INSERT INTO 课程 VALUES (12,'java',0.1) ;
INSERT INTO 学生选课 VALUES (1,10,99) ;
INSERT INTO 学生选课 VALUES (2,10,99) ;
INSERT INTO 学生选课 VALUES (1,11,80) ;
INSERT INTO 学生选课 VALUES (2,12,90) ;

所有的数据都可以进行各自的实体表的维护。

第三设计范式（一对多映射）

数据库表之中不存在非关键字段对任意一候选关键字段的传递函数依赖

示例讲解

雇员编号 姓名 职位 部门名称 位置 工资
1 张三 软工 开发02 天津 8000

存在有一个传递的函数依赖，通过张三可以知道他的部门名称，但是通过部门名称又可以找到部门位置

  • (姓名、职位) —> (部门名称) —> (部门位置)， 这属于一个传递关系

范例： 设计数据表，可以描述一个学校有多个学生

* 第一范式如果要使用的话，一定会出现传递依赖关系；
* 第二范式可以描述，但是也会存在有：一个学生在多个学校；
* 使用第三范式设计

Eg（错误设计）

    CREATE TABLE 学校-学生 (
        学校ID    NUMBER PRIMARY KEY ,
        学校名称    VARCHAR2(50) ,
        学校地址    VARCHAR2(50) ,
        学生姓名    VARCHAR2(50) ,
        学生年龄    NUMBER
    ) ;

Eg(modified)


CREATE TABLE 学校 (
    学校ID    NUMBER PRIMARY KEY ,
    学校名称    VARCHAR2(50) ,
    学校地址    VARCHAR2(50)
) ;
CREATE TABLE 学生 (
    学生ID    NUMBER PRIMARY KEY ,
    学生姓名    VARCHAR2(50) ,
    学生年龄    NUMBER ,
    学校ID    NUMBER REFERENCES 学校(学校ID)
) ;

总结：

  • 要想熟练掌握数据库分析与设计能力，是需要积累大量的项目开发经验，必须知道程序能做那些？数据库又可以做哪些事情。

  • 在实际开发之中，第三设计范式是最优先考虑的。

单 表 👉 第一范式

多对多 👉 第二范式

一对多 👉 第三范式

数据库设计综合实战

巩固数据库的设计范式。

设计要求

设计一个购物网站程序，基本要求：

1 . 所有的商品由后台管理员进行管理，所发布的商品必须经过其他管理员审核后才可以由顾客购买；


商品信息应该是一张实体表，实体表一定需要主键，由于此数据的主键不需要特殊的处理，所以可以直接使用序列生成。


商品如果要有品牌的话那么一个品牌就需要对应多个商品，使用第三范式


所有商品一定由管理员负责维护，所以需要一张管理员信息表。

所有发布的商品不能够直接让前台用户浏览，需要先审核，那么需要一个明确的字段来描述当前商品的状态eg：有一个flag字段，这个字段可以用来描述商品的状态


flag = 0: 此商品等待审核； flag = 1: 此商品审核通过可以正常浏览;
flag = 2: 此商品审核不通过需要重新提交审核; flag = 3: 此商品已经被逻辑删除了，不应该被搜索到;


如果真的需要审核的话，则审核管理员和发布管理员不能为同一个人，但是这一个判断是由程序的逻辑业务来进行处理的，所以本次只是提供数据的存储。

2 . 每一个管理员都有一个自己的角色，每一个角色包含有多个权限组，每个权限组可可以属于不同的角色，每个权限组有多个权限；


每一个角色可以有多个管理员具备，但是一个管理员只能够有一个角色，属于一对多关系，使用第三范式。
每一个角色包含有多个权限组，每个权限组可以属于多个角色，典型的多对多关系使用第二范式。

每一个权限组可以有多个权限，属于一对多关系，使用第三范式

3 . 所有的商品需要按照类型进行分类，分类分为一级类型与二级类型;


商品分为两类：类型与子类型，一个类型下有多个子类型，属于第三范式
商品也有所属类型，一个类型有多个商品，一个子类型下也有多个商品，属于第三范式




4 . 每一件商品都可以有多张展示图片；


一个商品可以有多张图片，属于一对多关系，图片先暂时使用BLOB类型描述




5 . 商品可以由用户进行评论，在评论时可以写出购买体会与打分；


每一个用户可以评价多个商品，每一个商品可以有多个用户评价，属于多对多关系，并且每一个用户针对每一个商品可以写出自己的心得体会以及打分



6 . 每一个用户在每次订单中可以购买多件商品，并且要保留有详细的购买记录；


一个用户可以有多个订单，每一个订单可以有多个购买的商品。每个订单购买的详细内容，可以设计一个订单详情表描述，即用户与订单属于第三范式，订单与订单详情属于第三范式

7 . 在进行购买商品配送时要求用户提供有详细收件地址、联系人、电话等信息；一个用户可以保存多个地址且可以设置一个常用地址；


如果要保存地址，需要知道用户所在的省份与城市，一个身份包含有多个城市信息，一对多关系，
一个省份可能有多个用户地址，或者说一个城市可能有多个用户地址
对于常用地址标记（flag=1, 为常用；flag=2为普通）

小结

  1. 数据库设计的前期一定要按照数据库的设计范式来完成；
  2. 数据库分析能力是一种长期积累的过程，量变–>质变，之后自然就知道那些地方可以进行调正优化了；
